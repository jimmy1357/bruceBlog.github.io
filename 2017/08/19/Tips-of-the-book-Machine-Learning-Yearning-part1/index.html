Tips of the Book "Machine Learning Yearning" ( Part Ⅰ )

Machine Learning
  • Machine Learning Yearning

PART 0 Preliminaries

1.随着数据规模的持续增加，使用NN比使用传统的ML算法更容易获得更好的性能。

（如何提升算法性能？(1) 训练更大的神经网络 (2) 使用更多的训练数据）






# PART Ⅰ Setting up development and test sets

2.训练集training set：用于训练算法；——训练不同模型

开发集dev set（调优集、Hold-out cross validation set）：用来调参、选择特征等；——判断哪个模型（算法）更优

测试集test set：评估算法性能，不参与调参和选择特征。——选择最优的模型，在测试集上evaluate

同时使用调优集和测试集可以让你的团队更快速的判断一个算法的性能好坏。换句话说，使用调优集和测试集的目的是指导你的团队让你的机器学习系统朝着更优的方向演进。（Gives you an unbiased estimate of performance.）

因此，你选择的dev set 和test set需要能够满足你未来实际数据的分布。

3.dev set 和 test set 应该具有相同的分布。

这样，当你的算法在dev set 上性能不错，但是在test set上性能较差时，那一定是你的算法在dev set上过拟合了，你需要扩充你的dev set。反之，当你的dev set和test set 分布不同时，出现上述情况，可能的原因就比较多了：

(1) 算法在dev set 上过拟合了.

(2) test set is harder than dev set， 也许你的算法其实不错，但是这个测试集不合适（比dev set复杂）.

(3) 测试集未必比调优集复杂，它只是和调优集不同，因此即便你在调优集上的性能再好，也只是浪费时间.

4.通常情况下，dev sets 数据量在1000到10000之间比较合适。

当然，当你的算法精度要求到0.01%时，你的dev set 数据量就要远大于10000条了，这样，你才能检测到算法0.01%的性能提升。

test sets的数据量如何选择？当你的总数据量在100到10000时，你可以选择20%~30%的数据作为测试集。但当你的数据规模更大时，20%的测试集就太多了。

一般，training set， dev set， test set的选择遵循如下规律：

Amount of data| training set(%) | dev set(%)| test set(%)
:—-:|:——:|:—-:|:—-:
100~10000| 60%| 20%| 20%
百万级以上| 98% or more| 1% or less| 1% or less

5.在某些情况下，我们看到数据集只包括training set 和test set（没有dev set），其实本质上来说，这里的test set 扮演的就是dev set的角色（参数调优）。但是这有一个缺陷，你的模型可能会在test set上过拟合。

6.使用单值评估（Single-number evaluation metric）衡量模型的优劣。例如准确率（Accuracy）。

如果你对精度和召回率情有独钟，可以使用F1 score（精度和召回率的几何平均），将这两个标准合一，变成一个单值评估标准。

回顾一下Precision 和 Recall的概念（拿文中的猫咪分类问题举例）：在dev 或 test set 中，

Precision： 在算法模型标注为猫的集合中，那些实际上就是猫图像的占比。

Recall：在整个dev或test set中实际是猫的集合中，有多少猫被算法模型成功标注的占比。

还有一种常用的将多值评估标准合一的方法：取多值的平均或者加权平均。

假如你对算法的准确率和执行时间都有要求，你可以这么做：

(1) 首先，定义一个可以接受的执行时间上限；

(2) 最大化准确率，但执行时间不超过你设定的上限；

在这个例子中，执行时间是“满意度标准”(satisficing metric)， 准确率是“优化标准”(optimizing metric)

假设你有N个标准来限定模型，你需要设置N-1个标准作为“满意度标准”，1个标准作为“优化标准”，

eg：

你想要做一个“唤醒程序”，例如，siri的“Hi siri”,来唤醒一个语音助手。你关注的是false positive rate（唤醒语有误，但仍然唤醒程序的 频率）以及 false negative rate（唤醒语正确，但没有唤醒程序的 频率）。

这两个标准如何统一？

你需要做的是，最小化false positive rate（优化标准），限制每天的false negative 出现频率不超过1.

对于一个新问题，事先知道什么样的模型可以达到最优效果是很难的，因此你所要做的就是不断迭代上述过程，寻找最优方法。

9.在哪种情况下，你需要更换调优/测试集，以及评估标准？

通常情况下，在一段较短的时间内（一周以内）提出一个初始版的调优/测试集以及评估标准。

It is better to come up with something imperfect and get going quickly, rather than over think this.

如果你发现你设置的初始评估标准和测试、调优集达到的效果与目标不符，务必尽快更改它！
例如，如果你的dev / test set + metric 显示你的分类算法A优于分类算法B，但你的团队认为分类算法B在你的产品中表现更优秀。这说明你该更换这些数据和标准了。

有如下三种情况，可能会导致dev / test set + metric 误将分类器A看做更优的分类器：

(1) 实际生产环境中的数据，与你的调优集、测试集数据的分布不符。（例如你的调优集、测试集中都是成年猫咪，而生产环境中的都是小猫）

(2) 在dev set 上过拟合了。这通常表现为：在dev set 上的分类效果远好于在test set 上。
如果你要跟踪团队的进展，你可以定期评估你的分类系统（使用test set），但是不要用test set来作为你的调优集，也不要使用测试集来决定你是否需要回滚分类器版本！否则，会让你的test set失去其评估的能力（模型将会在test set上过拟合）。

(3) 评估标准与你的优化方向不符。

在你的猫咪分类器中，使用现有的评估标准，结果是分类器A优于分类器B，但是你的分类器A概率性的会将一些黄暴图片识别为猫咪。尽管分类器A的准确度更高，但是它的副产品（黄暴图片）是让人无法忍受的。
情况是这样的，你的评估标准错误的认为分类器A优于B，而实际正好相反。因此，这个评估标准就需要更改了。例如，你可以设置惩罚参数作为满意度标准，惩罚错分类的黄暴图片。强烈建议你重新选择评估标准，而不是继续使用一个不靠谱的标准，手动选择分类器。
